1 About {#mainpage}
2 =====
3 
4 FreeNOS (Free Niek's Operating System) is an experimental microkernel based
5 operating system for learning purposes written in C++. You may use the code
6 as you wish under the terms of the GPLv3.
7 
8 Visit the project website at http://www.FreeNOS.org for more information.
9 
10 Features
11 ========
12 
13 * Intel x86 (PC) and ARMv6/ARMv7 architectures (Raspberry Pi 1,2,3)
14 * Virtual memory
15 * Simple task scheduling
16 * Inter Process Communication (IPC)
17 * Symmetric Multi Processing with MPI support (Intel x86 only)
18 * Devices:
19  * VGA/Keyboard consoles (also supported by Ed's libteken (http://80386.nl/projects/libteken/)
20  * i8250 serial UART
21  * PCI host controller
22  * CMOS RTC clock
23  * ATA host controller
24  * USB controller and (root)hub (Raspberry Pi only)
25  * Loopback network and SMSC95xx ethernet (Raspberry Pi only)
26 * Filesystems:
27  * Virtual file system (VFS)
28  * Temporary file system (TmpFS)
29  * Linnenbank file system (LinnFS)
30 * Networking (IP, UDP, ICMP, for Raspberry pi 1 only)
31 * POSIX, ANSI C libraries
32 * Dynamic and Shared memory
33 * Fully automatic autotester
34 * All sources documented with Doxygen (http://www.doxygen.org)
35 * User and kernel code written from scratch in C++
36 * Very small microkernel (~2K lines of C++ code including a tiny part in assembly)
37 * Builds with recent GCC (http://gcc.gnu.org), LLVM (http://www.llvm.org) and SCons (http://www.scons.org) versions on POSIX systems
38 
39 Host Setup
40 ==========
41 
42 First install all required build dependencies. FreeNOS needs SCons, an C++ compiler and for Intel targets a tool to generate ISO images.
43 Follow the instructions below to install the build dependencies on your host OS.
44 
45 *Ubuntu*
46 
47 Update your system repository cache and install the required development tools using:
48 
49  $ sudo apt-get update
50  $ sudo apt-get install build-essential scons genisoimage xorriso qemu-system
51 
52 If your Ubuntu host is 64-bit, you need to install the GCC multilib package
53 to cross compile for the 32-bit architecture:
54 
55  $ sudo apt-get install gcc-multilib g++-multilib
56 
57 Alternatively, you can install the LLVM/Clang compiler as well:
58 
59  $ sudo apt-get install clang
60 
61 *FreeBSD*
62 
63 Update your system repository cache and install the required development tools using:
64 
65  % su -
66  # pkg update
67  # pkg install qemu scons cdrkit-genisoimage xorriso gcc
68 
69 On FreeBSD, make sure that the latest version of the GNU linker (from pkg) is used:
70 
71  # mv /usr/bin/ld.bfd /usr/bin/ld.bfd.orig

73 Building FreeNOS
74 ================
75 
76 To download and extract the released source code on your host OS, run the
77 following commands. Replace 'x.x.x' with the version of FreeNOS:
78 
79  $ wget http://www.FreeNOS.org/pub/FreeNOS/source/FreeNOS-x.x.x.tar.gz
80  $ tar zxf FreeNOS-x.x.x.tar.gz
81 
82 To get the latest development source, you can clone the GIT archive:
83 
84  $ git clone https://github.com/nieklinnenbank/FreeNOS
85 
86 To build FreeNOS using default settings (Intel, using GCC with debugging enabled), run:
87 
88  $ scons
89 
90 To build FreeNOS with all full build commands printed on the console, set the
91 VERBOSE build variable to True:
92 
93  $ scons VERBOSE=True
94 
95 To build FreeNOS with compiler optimizations enabled, set DEBUG to False.
96 Note that will make debugging using GDB more difficult:
97 
98  $ scons DEBUG=False
99 
100 Instead of providing build variables on the command line, you can
101 also change the 'build.conf' configuration file for the target. The build configuration
102 file contains build variables, such as compiler flags and parameters for the target.
103 Similary, the 'build.host.conf' file contains build variables for the host OS programs.
104 See the example build.conf and build.host.conf files for more details.
105 
106 Additionally, any environment variables set in the shell will be automatically converted
107 to identical build variables when running scons.
108 
109 To build for the Raspberry Pi 2, copy the example build configuration file and run SCons:
110 
111  $ cp config/arm/raspberry2/build.conf .
112  $ scons
113 
114 To cleanup your build directory, use:
115 
116  $ scons -c
117 
118 Or use the following command to also remove temporary SCons files:
119 
120  $ rm -rf build .sconf_temp .sconsign.dblite

122 Running FreeNOS
123 ===============
124 
125 To run the autotester of FreeNOS on the host OS, run:
126 
127  $ scons test
128 
129 To run the same autotester on FreeNOS under Qemu, run:
130 
131  $ scons qemu_test
132 
133 To start FreeNOS in a Qemu virtual machine with a serial console,
134 use the following command:
135 
136  $ scons qemu
137 
138 To debug FreeNOS using GDB, you need to have build using the DEBUG=False build variable.
139 Ensure that your host OS has GDB available for debugging the target architecture (Intel or ARM).
140 For Ubuntu:
141 
142  $ sudo apt-get install gdb-multiarch
143 
144 Open two terminals. In the first terminal, launch FreeNOS in Qemu with the internal gdbserver enabled:
145 
146  $ scons qemu_debug
147 
148 In the second terminal, start GDB and connect to the Qemu internal gdbserver using:
149 
150  $ gdb-multiarch ./build/intel/pc/kernel/intel/pc/kernel
151  (gdb) target remote localhost:1234
152 
153 You can now use standard GDB commands to interactively debug the FreeNOS kernel for intel.
154 Similarly, you can also debug a user program (./build/intel/pc/bin/XXX) or the FreeNOS
155 kernel for ARM (./build/arm/raspberry2/kernel/arm/raspberry2/kernel).
156 
157 intel/pc
158 --------
159 
160 For Intel targets, you can also launch Qemu using the graphical VGA console (using SDL):
161 
162  $ scons qemu_sdl
163 
164 To test FreeNOS on real hardware, write the boot ISO to a bootable device, such as an USB disk.
165 Insert the USB disk to your host OS and find the appropriate device name:
166 
167  $ fdisk -l
168 
169 Alternatively, use the dmesg command to find the device name. Write the boot ISO
170 using the following command, where sdX is the name of the USB disk device.
171 Note: make sure to select the correct device to prevent data loss:
172 
173  $ sudo dd if=build/intel/pc/boot.iso of=/dev/sdX
174  $ sudo sync
175 
176 Insert the USB disk device to the target system and make sure to choose it as
177 boot device when the computer starts (e.g. via BIOS). You should see the FreeNOS VGA console.

179 arm/raspberry
180 -------------
181 
182 To build FreeNOS for running on the Raspberry Pi 1, use the provided build.conf file
183 and rebuild the system:
184 
185  $ cp config/arm/raspberry/build.conf .
186  $ scons
187 
188 To test FreeNOS for the raspberry pi 1 under Qemu, rebuild the system
189 with the QEMU_BUILD build variable set to True:
190 
191  $ scons QEMU_BUILD=True
192  $ scons qemu
193 
194 To test on a real Raspberry Pi using SD card, download the latest Raspbian 'Lite' image
195 from https://www.raspberrypi.org/downloads/raspbian/ and unzip it. Insert the SD card to
196 your reader/writer slot on your host OS and find the appropriate device name using:
197 
198  $ fdisk -l
199 
200 Alternatively, use the dmesg command to find the device name. Write the Raspbian image to the
201 SD card using the following command, where sdX is the name of the SD card device.
202 Note: make sure to select the correct device to prevent data loss:
203 
204  $ sudo dd if=/path/to/raspbian/image.img of=/dev/sdX
205  $ sudo sync
206 
207 Re-insert the SD-card to auto mount it on your host OS. Copy the FreeNOS raspberry pi kernel executable
208 to the '/boot' partition on the Raspbian image using (on Ubuntu):
209 
210  $ sudo cp build/arm/raspberry/kernel.img /media/boot/kernel.img
211  $ sudo sync
212  $ sudo umount /media/boot
213 
214 You will need a USB-to-TTL-serial cable to connect to the Raspberry Pi UART.
215 Any USB-to-TTL-serial cable will work, for example the Adafruit USB-to-TTL-serial cable:
216 
217  https://www.adafruit.com/product/954
218 
219 Connect your USB-to-serial cable to the Raspberry Pi using the GPIO pins:
220 
221  [ ] [ ]
222  [ ] [X] Pin 4 (V5) <--- Red (only needed if powered via USB)
223  [ ] [X] Pin 6 (Ground) <--- Black
224  [ ] [X] Pin 8 (RX) <--- White
225  [ ] [X] Pin 10 (TX) <--- Green
226  [ ] [ ]
227  [ ] [ ]
228  [ ] [ ]
229  [ ] [ ]
230  [ ] [ ]
231 
232 Also see the following tutorial by Adafruit on connecting the serial cable to the Raspberry Pi:
233 
234  https://learn.adafruit.com/adafruits-raspberry-pi-lesson-5-using-a-console-cable/connect-the-lead
235 
236 Connect your Raspberry Pi to the host to power it via USB or use an external power adapter.
237 On the host OS, launch a serial console client program such as minicom and connect it to the
238 USB serial device (baudrate 115200):
239 
240  $ minicom -D /dev/ttyUSB0
241 
242 You should now see the FreeNOS console output.

244 arm/raspberry2
245 --------------
246 
247 Follow the same instructions as for the Raspberry Pi 1 above, but use the Raspberry Pi 2
248 configuration file instead to build FreeNOS:
249 
250  $ cp config/arm/raspberry2/build.conf .
251  $ scons
252 
253 When copying the kernel executable to the SD card '/boot' partition, use the 'kernel7.img' filename
254 to make sure the Raspberry Pi bootloader correctly starts the kernel (ARMv7):
255 
256  $ cp build/arm/raspberry/kernel.img /media/boot/kernel7.img
257 
258 Additionally, for raspberry pi 2 and newer, add the following to the /boot/config.txt
259 file to ensure the first UART is available on GPIO pins 8 and 10:
260 
261  dtoverlay=pi3-miniuart-bt
262  enable_uart=1

264 Using FreeNOS
265 =============
266 
267 When FreeNOS starts the system will print bootup output and present the system
268 login prompt. Currently the login will accept any username value and does not
269 ask for a password. The FreeNOS interactive console prompt looks like the following:
270 
271  (localhost) / #
272 
273 FreeNOS has a UNIX-like interface and you may enter any of the commands available
274 in the /bin directory:
275 
276  (localhost) / # ls bin
277 
278 For example, the 'ps' command prints a list of all processes running in the system:
279 
280  (localhost) / # ps
281 
282 You can also run the fully automatic autotester inside Qemu or on real hardware
283 using the following command:
284 
285  (localhost) / # /test/run
286 
287 To view some information about the FreeNOS version and hardware settings
288 you can use the 'sysinfo' command:
289 
290  (localhost) / # sysinfo
291 
292 The '/' in the prompt indicates the current active directory. Change it with the 'cd'
293 built-in shell command:
294 
295  (localhost) / # cd /tmp
296  (localhost) /tmp #
297 
298 The FreeNOS shell has several built-in commands, use the 'help' command
299 to view all the built-in shell commands:
300 
301  (localhost) / # help
302 
303 Example application program for calculating prime numbers is the /bin/prime command.
304 To let it compute all prime numbers up to 1024 and output the prime number results use:
305 
306  (localhost) / # prime --stdout 1024
307 
308 For Intel, the prime command also has a MPI variant called 'mpiprime' which can
309 compute the prime numbers in parallel using multiple cores. To run it and let the
310 shell measure the time taken use:
311 
312  (localhost) / # time mpiprime 2000000
313 
314 You can compare the time result versus the time take of the single core program
315 where it computes the same number of primes:
316 
317  (localhost) / # time prime 2000000

319 Jenkins Continuous Integration
320 ==============================
321 
322 Master Setup
323 ------------
324 
325 Install Jenkins on your host OS using your favorite package manager or from the official website (https://jenkins.io/).
326 Follow the installation wizard instructions and after installation go to the Jenkins web interface at: http://localhost:8080
327 
328 After installation, navigate to: Manage Jenkins > Manage Plugins
329 Make sure the following plugins are installed. Choose the plugins from the 'Available' tab to find the plugins
330 which are not yet installed:
331 
332 - Matrix Project Plugin (https://wiki.jenkins-ci.org/display/JENKINS/Matrix+Project+Plugin)
333 - Matrix Combinations Plugin (https://wiki.jenkins-ci.org/display/JENKINS/matrix+combinations+plugin)
334 - Node and Label Parameter Plugin (https://wiki.jenkins-ci.org/display/JENKINS/NodeLabel+Parameter+Plugin)
335 - Libvirt Slaves Plugin (https://wiki.jenkins.io/display/JENKINS/Libvirt+Slaves+Plugin)
336 - SSH Plugin (https://wiki.jenkins-ci.org/display/JENKINS/SSH+plugin)
337 - Workspace Cleanup Plugin (https://wiki.jenkins-ci.org/display/JENKINS/Workspace+Cleanup+Plugin)
338 
339 After finishing Jenkins configuration, install KVM and Libvirt on your host OS. For Ubuntu use:
340 
341  $ sudo apt-get install qemu-kvm libvirt-bin virtinst
342 
343 Connect Jenkins to libvirt via SSH, navigate to: Manage Jenkins > Configure System
344 Choose 'Add a new cloud' and select 'Hypervisor Type': QEMU,
345 'Hypervisor Host': localhost and choose your desired SSH username.
346 
347 Press 'Advanced' to choose authentication parameters and press 'Test Connection' to
348 verify Jenkins can access libvirt via SSH. For Ubuntu, you need to ensure the SSH user
349 is in the 'kvm' and 'libvirtd' groups:
350 
351  $ sudo usermod -a -G kvm,libvirtd jenkins
352 
353 Jenkins need to find the KVM guests by their hostname. You can either manually configure
354 the DNS hostname to IP translation in the /etc/hosts file or automatically using the
355 internal libvirt DNSMasq server.
356 
357 To setup automatic DNS translation in KVM, first edit the 'default' libvirt network interface
358 to add the '.kvm' local-only domain:
359 
360  $ virsh net-edit default
361 
362 Add the following line inside the <network>....</network> tag:
363 
364  <domain name='kvm' localOnly='yes'/>
365 
366 To use the libvirt internal DNSMasq server on your host, you may change the /etc/resolv.conf
367 file to add the following:
368 
369  nameserver 192.168.122.1
370 
371 If your host OS uses NetworkManager for networking, add the following files:
372 
373  $ sudo -s
374  # cat > /etc/NetworkManager/conf.d/localdns.conf
375  [main]
376  dns=dnsmasq
377 
378  # cat > /etc/NetworkManager/dnsmasq.d/libvirt_dnsmasq.conf
379  server=/kvm/192.168.122.1
380 
381 Ensure that all KVM guests have their hostname set in the .kvm domain (e.g. someguest.kvm).
382 Restart libvirt and optionally NetworkManager to apply the changes:
383 
384  $ sudo /etc/init.d/libvirt-bin restart
385  $ sudo /etc/init.d/NetworkManager restart
386 
387 You should now be able to resolve the hostnames of your guests in the .kvm domain after they
388 are started, for example:
389 
390  $ virsh start ubuntu-1804
391  $ host ubuntu-1804.kvm
392 
393 Also visit the following page for more details on this automatic DNS setup for KVM:
394 https://liquidat.wordpress.com/2017/03/03/howto-automated-dns-resolution-for-kvmlibvirt-guests-with-a-local-domain/

396 FreeBSD 12.0 Slave
397 ------------------
398 
399 Run the example installation script in ./support/jenkins/freebsd-12.sh from the FreeNOS sources
400 to setup the KVM guest with FreeBSD 12.0. Also see the comments in the installation script for more info:
401 
402  jenkins@host$ cd support/jenkins
403  jenkins@host$ ./freebsd-12.sh
404 
405 To configure the FreeBSD 12.0 slave in Jenkins for building FreeNOS, you first need to
406 make sure the Jenkins user can login to the slave using SSH. Optionally, you can
407 configure this with public key authentication:
408 
409  jenkins@host$ virsh start freebsd-12
410  jenkins@host$ ssh-keygen
411  jenkins@host$ ssh-copy-id jenkins@freebsd-12.kvm
412 
413 Test if the Jenkins user can login to the FreeBSD KVM guest with SSH (via password or key):
414 
415  jenkins@host$ ssh jenkins@freebsd-12.kvm
416 
417 Copy the example node configuration XML file to the Jenkins installation directory:
418 
419  jenkins@host$ mkdir /var/lib/jenkins/nodes/freebsd-12-test
420  jenkins@host$ cp freebsd-12.node.xml /var/lib/jenkins/nodes/freebsd-12-test/config.xml
421 
422 Also copy the job configuration XML file to the Jenkins installation directory:
423 
424  jenkins@host$ mkdir /var/lib/jenkins/jobs/FreeNOS-freebsd12-test
425  jenkins@host$ cp freebsd-12.job.xml /var/lib/jenkins/jobs/FreeNOS-freebsd12-test/config.xml
426 
427 Restart the Jenkins server to use the new configuration files:
428 
429  $ sudo /etc/init.d/jenkins restart
430 
431 Jenkins still needs authentication credentials to connect to the slave. Navigate to:
432 
433  Manage Jenkins > Manage Nodes > freebsd-12 > Configure
434 
435 In the sub-menu 'Secondary launch method', under 'Launch agents via SSH',
436 select existing SSH credentials to let jenkins use it to login to the slave node
437 or add a new credentials with the 'Add' button. If you generated an SSH key for
438 the jenkins user on the host OS, insert the SSH private key or provide username/password.
439 
440 Press the 'Save' button to finish and use 'Launch Agent' to test the connection.
441 
442 You can now press the 'Schedule a Build' button on the 'FreeNOS-freebsd12' build job to test.
